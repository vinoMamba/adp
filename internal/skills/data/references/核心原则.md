# 核心原则（所有 ADP skill 共享）

本文件是所有 ADP skill 的共享不变量。每个 skill 在开始工作前必须先读这份文件，再按需读取其它 `references/`。

## 目录结构

每个客户工作区固定如下：

- `原始资料/`：保存公开调研、拜访纪要、方案报价、CRM记录、系统资料，**不改写原文**。
  - `原始资料/公开调研/`、`原始资料/拜访纪要/`、`原始资料/方案报价/`、`原始资料/CRM记录/`、`原始资料/系统资料/`
- `客户知识库/`：保存 agent 维护的结构化事实、判断、假设和行动计划。
- `输出/`：只保存一份持续迭代的 `客户名称-ADP.md`。
- `metadata.json`：客户级元数据（name/owner/stage/status/created/updated/model/materials_count）。

## 沉淀与输出原则

- 原始资料不改写，统一放入 `原始资料/`。
- 结构化判断写入 `客户知识库/`。
- **一个客户只有一份 ADP 输出产物**：`输出/客户名称-ADP.md`。
- 后续更新必须迭代 `输出/客户名称-ADP.md`，不要创建 `ADP-日期.md`、`ADP-v2.md`、阶段版或多份备份。
- ADP 历史变化记录在 `客户知识库/更新日志.md`，不是通过复制多份输出文件实现。

## 事实 / 判断 / 假设

- 事实写来源，判断写依据，假设写待验证。
- Grounding 优先：`客户知识库/*.md`、`输出/*-ADP.md`、`更新日志.md` 是中间层汇总；预算、成本、账号数、金额、日期、角色、阶段等高风险事实必须回溯到最新、最权威的原始资料或工具结果（见 `references/事实溯源与Grounding.md`）。
- 不编造联系人、预算、采购流程、客情强度、竞品状态和客户态度。

## 状态写入约定（重要）

skill 永远不直接写以下文件，只通过 `adp` CLI 回调：

| 状态 | 写入命令 | 说明 |
|---|---|---|
| `metadata.json` | `adp status <客户> [--stage] [--state] [--model]` | 更新阶段 / 状态 / 模型 |
| `客户知识库/更新日志.md` | `adp log <客户> --action <动作> [--judgement <判断>]` | 追加更新记录 |
| `客户知识库/来源登记.md` | `adp source <客户> --origin <来源> [--type] [--date] [--authority] [--page] [--key] [--note]` | 登记新资料 |
| 客户工作区脚手架 | `adp create <客户> [--owner] [--stage]` | 新建工作区（含全部目录 + 模板 + metadata） |

CLI 是这些状态的唯一写入者。`输出/客户名称-ADP.md` 和 `客户知识库/` 下其它知识页（客户画像、现状、人物与决策链、机会与动机、行动计划、索引）由 skill 直接编辑。

`metadata.json` 的 status 字段取值：`draft`（新建）→ `updating`（正在迭代）→ `ready`（ADP 已生成且最新）；另有 `stale`（新资料进来未重新生成）。

## 知识页职责

| 页面 | 职责 |
|---|---|
| `客户画像.md` | 客户简介、业务、行业地位、经营背景、上下游、子公司 |
| `现状.md` | 现网系统、已购买的钉钉产品、客户群、项目对接记录、数字化现状 |
| `人物与决策链.md` | 组织架构、关键人、核心 KP、决策链、EB/UB/TB/Coach、客情等级积分 |
| `机会与动机.md` | PPL机会、痛点、购买动机、购买标准、机会质量五维评估、价值假设、为什么跟我们买 |
| `行动计划.md` | 五大关键行为、团队作战、产品线近期规划、风险与应对、待验证问题 |
| `来源登记.md` | 每份资料的来源、类型、日期、权威层级、影响页面和关键字段索引（由 `adp source` 维护） |
| `更新日志.md` | 每次 wiki 更新的时间、动作和判断变化（由 `adp log` 维护） |

## 输出要求

标准 ADP 的十段结构和第五节子段要求见 `references/ADP方法论.md`。生成或重构 `输出/客户名称-ADP.md` 时使用 `references/标准ADP输出模板.md` 中的骨架。

Mermaid 图表使用 `references/标准ADP输出模板.md` 的兼容示例：优先使用基础 `graph` / `flowchart`，不要使用 `quadrantChart`、`x-axis`、`y-axis` 等新语法；不要在节点里放 emoji、示例图标、独立图例图或示例线条图，状态用文字标签和节点样式表达。

最终 ADP 输出与内部知识页分层处理：

- `客户知识库/机会与动机.md` 保留机会质量五维评估底表，供判断和回源使用；最终 ADP 第七节只输出产品线近期规划简表。
- `客户知识库/人物与决策链.md` 保留客情判断依据；最终 ADP 第五节的「决策链客情分析」不画图，只输出简表。
- `客户知识库/*`、`来源登记.md` 和原始资料承担溯源职责；最终 ADP 只呈现结论、判断和规划，不体现来源字段、来源链接、页码或"待回溯原始来源"等溯源表述。

## 质量门槛

- 不改变 `输出/客户名称-ADP.md` 的十段主结构。
- 高风险事实不能只引用 `客户知识库/*.md`、`输出/*-ADP.md` 或 `更新日志.md`；必须引用原始资料、系统导出、用户本轮工具结果或公开权威来源。找不到原始来源时写"待回溯原始来源"或"待确认"。
- 最终 ADP 不体现数据来源，不附来源链接、来源名称、页码、截图位置或来源文件名；这些只维护在 wiki 和原始资料体系里。
- 拜访信息写事实和判断即可；日期、参与人和来源文件维护在 wiki，不在最终 ADP 单独展示来源。
- 第二节「人员关系图谱」必须包含关系图和图下总结段，不能只有图。
- 第三节「数字化现状」必须落成 Markdown 表格，不能只写字段提示。
- 未接触人物不能写强客情结论。
- 客情分析必须对照决策链人员逐人评分，使用 `-3` 到 `+3` 的客情等级积分；证据不足时写"待确认"，不要硬判。
- 第五节「决策链客情分析」不输出图表，只输出这 5 列：客户姓名、是否接触、决策力 / 客户关系、角色识别（可多选）、对钉钉支持程度（客情等级积分）。
- 客情等级积分必须有活动空间、信息质量、立场程度或客户行为依据；有明确项目时写项目进展+客户关系进展，没有明确项目时写客户关系进展。
- 有明确 PPL、销售目标、报价或推进中项目时，必须维护 `客户知识库/机会与动机.md` 中的「机会质量五维评估」：客户业务及行业分析、使用部门参与决策、产品解决方案匹配度、预计金额及对应决策者、预计时间关联紧迫度。
- 第七节「近期规划（产品线）」最终只输出一个表格，字段固定为：计划时间、产品、对接部门。
- 第八节「已购买产品」只填写客户已经购买的钉钉产品，不写其他厂商产品。
- 行动计划必须包含目标人物、目的、负责人、时间和验证方式。
- Mermaid 图表必须能在常见 Markdown / Obsidian 预览中渲染，避免依赖兼容性不稳定的 Mermaid 新图表类型。
