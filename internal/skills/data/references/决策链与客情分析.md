# 决策链与客情分析

## 核心 KP 个人判断

核心 KP 不是职位最高的人，而是当前项目中最能推动下一步的人。判断维度：

- 是否直接主导当前议题。
- 是否能协调多个部门。
- 是否能影响预算、技术、采购或拍板。
- 是否愿意给我方信息、资源或内部路径。
- 是否与我方已有有效接触。

推荐写法：

```markdown
## 核心 KP 个人

- 当前判断：
- 依据：
- 风险：
- 下一步验证动作：
```

## 决策角色

| 角色 | 定义 |
|---|---|
| 发起人 | 提出需求或推动议题出现的人 |
| 使用推动者 | 代表实际使用部门推动落地的人 |
| 技术把关者 | 判断安全、集成、部署、数据、运维可行性的人 |
| 预算把关者 | 能影响预算申请、费用合理性和付款安排的人 |
| 采购把关者 | 影响供应商准入、招采流程和商务条款的人 |
| 最终拍板者 | 能对项目成败做最终决定的人 |
| 关键影响者 | 无正式拍板权，但能显著影响内部意见的人 |

## EB / UB / TB / Coach

| 角色 | 判断标准 |
|---|---|
| EB | 经济买家，影响预算和最终商业决策 |
| UB | 使用买家，代表业务部门和使用场景 |
| TB | 技术买家，关注集成、安全、运维、数据和部署 |
| Coach | 愿意帮助我方理解客户内部路径并指导行动的人 |

## 客情等级积分

客情分析统一使用 `-3` 到 `+3` 的积分。积分必须同时参考活动空间、信息质量和立场程度，不能只凭主观感觉打分。

| 客情等级积分 | 活动空间 | 信息质量 | 立场程度 |
|---|---|---|---|
| -3 分 | 工作空间 | 将信息传递给友商 | 极力反对 |
| -2 分 | 工作空间 | 提供误导信息 | 负向引导 |
| -1 分 | 工作空间 | 提供公开信息 | 倾向友商 |
| 0 分 | 工作空间 | 提供公开信息 | 保持中立 |
| 1 分 | 社交空间 | 提供有效信息 | 倾向我方 |
| 2 分 | 半私人空间 | 被动提供私密信息 | 指导行动 |
| 3 分 | 私人空间 | 主动提供私密信息 | 强力推荐 |

证据不足时写“待确认”，不要硬给分。若三类证据不完全一致，以最能影响项目推进的客户行为为主，并在依据中说明原因。

## 客情分析规则

- 区分已接触和未接触。
- 区分个人关系、组织关系和项目关系。
- 区分客户主动反馈和我方推断。
- 每个强判断都必须写依据。
- 决策链图上的每个人都必须进入客情评分表，不能只写支持者。
- 最终 ADP 的「决策链客情分析」不放图，只输出简表。
- 最终 ADP 的客情评分表固定为 5 列：客户姓名、是否接触、决策力 / 客户关系、角色识别（可多选）、对钉钉支持程度（客情等级积分）。
- 有明确项目时，每个人都要写「项目进展 + 客户关系进展」；没有明确项目时，写客户关系进展，并合并到“决策力 / 客户关系”一列中表达。
- `客户知识库/人物与决策链.md` 可以保留更细的依据与下一步动作，但输出到 ADP 时要收口成简表。
- 客情结论必须落到下一步动作，而不是停留在描述。

## Mermaid 图表表达规则

- 决策链图只使用基础 `graph` / `flowchart` 语法，避免 `quadrantChart`、`x-axis`、`y-axis` 等兼容性不稳定的新图表类型。
- 节点标签使用文字表达状态，不使用 emoji、示例图标、独立图例图或示例线条图。
- 客情分析以表格为主，不再输出客情象限图。
